# Monitoring Research for Mac Mini

This note summarizes practical monitoring options for a freshly bootstrapped Mac mini and how they map to common Google Cloud observability patterns.

## Goal

Track enough host activity to answer:

- Is the machine alive and healthy?
- What software is installed and running?
- What commands/processes executed?
- Who logged in, used sudo, changed config, or accessed sensitive services?
- Which important files changed?
- Which cloud resources were accessed, allowed, or denied?

## Important Constraint

Google Cloud Ops Agent is the default agent for many Google Cloud VM observability setups, but it is not the right direct fit for a Mac mini. Google documents Ops Agent support for Compute Engine/Bare Metal Solution style environments and standard Linux/Windows VM operating systems, not macOS/on-prem machines.

For Mac mini, use a comparable architecture:

1. Local collectors on macOS.
2. A log forwarder.
3. Cloud Logging buckets, sinks, metrics, alerts, and dashboards.

## macOS Sources

### Built-in macOS logs

macOS uses Unified Logging. Locally, Apple Console can inspect log messages, activities, and reports generated by the computer and connected devices.

Useful commands:

```bash
log show --last 1h --style json
log stream --style json
log show --predicate 'eventMessage CONTAINS[c] "sudo"' --last 24h
```

Tradeoff: Unified Logging is not a simple append-only file that a log shipper can tail. For cloud export, run a small launchd job that periodically exports selected logs to JSONL files, then forward those files.

### Traditional files

Useful local paths:

```text
/var/log/system.log
/var/log/install.log
/var/log/wifi.log
/var/log/fsck_*.log
/Library/Logs/
~/Library/Logs/
```

These are easier to tail, but they do not cover all modern macOS activity.

### Shell and bootstrap history

For this bootstrap project, add a transcript log around setup execution:

```bash
mkdir -p "${HOME}/Library/Logs/macos-bootstrap-kit"
./bootstrap.sh 2>&1 | tee -a "${HOME}/Library/Logs/macos-bootstrap-kit/bootstrap.log"
```

For interactive shells, history is useful but not authoritative. Users can disable, edit, or bypass shell history.

### osquery

osquery is the best low-friction host visibility layer for this use case. It exposes OS state as SQL tables and can run as a daemon with scheduled queries.

Good scheduled query categories:

- running processes
- process execution events
- logged-in users
- users and groups
- launchd services
- listening ports and established sockets
- installed apps and Homebrew packages
- browser extensions
- file integrity monitoring for selected paths

Useful paths to monitor:

```text
/Applications
/Library/LaunchAgents
/Library/LaunchDaemons
/usr/local/bin
/opt/homebrew/bin
~/.ssh
~/.zshrc
~/.gitconfig
```

Tradeoff: detailed process/file eventing may require privacy permissions such as Full Disk Access and careful configuration. File access auditing at "who read this exact file" granularity is not something macOS exposes cleanly for all files without deeper Endpoint Security/EDR tooling.

## Cloud Logging Pattern

Google Cloud's default observability pattern is:

- Collect logs and metrics with an agent where supported.
- Store logs in Cloud Logging buckets.
- Query in Logs Explorer.
- Create log-based metrics.
- Create alerting policies.
- Route logs with sinks to BigQuery, Cloud Storage, or Pub/Sub.
- Use Cloud Audit Logs to answer "who did what, where, and when" for Google Cloud resources.

Cloud Audit Logs categories:

- Admin Activity: written by default for resource/config changes.
- System Event: written by default for Google-system changes.
- Policy Denied: generated by default for denied access due to security policy.
- Data Access: usually disabled by default except BigQuery; enable explicitly for sensitive services.

## Recommended Mac Mini Architecture

### Baseline

Install:

```text
osquery
fluent-bit
google-cloud-sdk
```

Keep these optional in the bootstrap until the Google Cloud project, service account, and retention policy are chosen.

### Local collection

Use:

- `osqueryd` for inventory, process, network, login, and file integrity events.
- A launchd job that exports selected Unified Logging entries to JSONL.
- A bootstrap transcript log written by `tee`.
- Optional local health script that writes JSON every minute: disk, memory, uptime, Docker status, Homebrew outdated count.

### Forwarding to Google Cloud

Use Fluent Bit's Stackdriver output plugin:

- Tail osquery result logs.
- Tail exported macOS Unified Logging JSONL.
- Tail bootstrap and maintenance logs.
- Send to Cloud Logging with resource type `generic_node`.
- Use labels like `host=mac-mini`, `env=home-lab`, `role=developer-workstation`.

### Cloud-side defaults

In Google Cloud:

- Create a dedicated project for workstation telemetry.
- Create a service account with the minimum logging write role.
- Store service account credentials in a protected path on the Mac, not in the repo.
- Use Cloud Logging buckets with explicit retention.
- Create sinks:
  - BigQuery for searchable security history.
  - Cloud Storage for long-term archive if needed.
  - Pub/Sub if later feeding SIEM/automation.
- Enable Data Access audit logs only for services where the extra volume is justified.
- Add log-based metrics and alerts for:
  - host heartbeat missing
  - repeated sudo failures
  - new launch daemon/agent
  - new listening port
  - SSH config/key changes
  - Docker daemon stopped
  - disk free below threshold
  - Cloud IAM policy changes
  - denied Google Cloud access

## Practical Bootstrap Additions

Do not enable cloud export blindly in the first bootstrap version. Add it behind config flags:

```bash
ENABLE_MONITORING="false"
GCP_PROJECT_ID=""
GCP_LOCATION="europe-west3"
MONITORING_NODE_ID="$(scutil --get LocalHostName 2>/dev/null || hostname)"
```

Then add a separate script:

```text
scripts/setup-monitoring.sh
```

That script should:

1. Install `osquery`, `fluent-bit`, and `google-cloud-sdk`.
2. Create `/usr/local/etc/macos-bootstrap-kit/monitoring`.
3. Refuse to continue unless a service-account JSON path is supplied.
4. Generate osquery config.
5. Generate Fluent Bit config.
6. Install launchd plists.
7. Start services.
8. Emit a test log entry and show the Cloud Logging query to verify ingestion.

## Recommended First Implementation

Start with three log streams:

1. `bootstrap.log`: exact setup transcript.
2. `osqueryd.results.log`: host inventory and security deltas.
3. `macos-unified-selected.jsonl`: filtered Unified Logging export for login, sudo, security, launchd, ssh, and software install events.

This gives useful observability without pretending a Mac mini is a Compute Engine VM.
